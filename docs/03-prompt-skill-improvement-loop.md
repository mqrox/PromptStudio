# Prompt Skill 改善サイクル

## なぜ保存するのか

PromptStudio では、生成されたプロンプト本文だけを保存しても不十分である。

重要なのは、以下が後から分かること。

- ユーザーは何を作りたかったのか
- どの会話で意図が具体化されたのか
- どのskillが使われたのか
- そのskillの情報源、バージョン、最終確認日は何か
- どのknowledgeが参照されたのか
- どのモデルや生成サービスを想定したのか
- 入力画像、入力動画、参照素材に何を使ったのか
- 生成結果が成功だったのか、失敗だったのか
- 失敗した場合、原因はプロンプト、入力素材、モデル選択、設定、skillのどこにありそうか
- 管理者が何を改善すべきと判断したのか

これらが残ることで、個人の試行錯誤が会社のskill改善につながる。

重要なのは、社員がすでに各自のAIチャットや生成モデルでプロンプトを作っている前提で設計すること。PromptStudio は「プロンプトを書いてくれるAI」ではなく、各自でばらばらになっている前提、knowledge、skill、モデル理解を揃え、レビュー可能な形で残す仕組みである。

このskill運用は、全員に同じようなプロンプトを書かせたり、個性のない平均的な出力へ寄せたりするためのものではない。目的は、各クリエイターの意図を活かすために、モデルへ渡す土台の前提条件を揃えることである。

## skillとは何か

PromptStudio における skill は、単なるプロンプトテンプレートではない。

skill は、以下を含む実務ノウハウの単位である。

- 画像生成向けの意図分解
- 動画生成向けの時間、動き、カメラ指定
- 広告、映像、CG / VFX、デザイン制作で必要な画作りの観点
- モデルごとの得意不得意
- モデルごとに必要な前提、入力形式、指定の粒度
- ネガティブプロンプトの考え方
- 案件ルールや禁止事項
- レビューでよく指摘される失敗パターン
- 出力を再現しやすくする記述順序
- 公式ガイドラインから取り込むべき注意点
- 社内の成功 / 失敗蓄積から改善された判断基準
- 表現を縛らず、前提条件の抜け漏れを防ぐための確認観点

## knowledgeとは何か

knowledge は、案件や会社で共有する文脈である。

例。

- 案件ごとのトーン、世界観、禁止事項
- ブランドガイドライン
- キャラクター設定
- 用語集
- 参照作品の分析メモ
- モデル別の使い方
- 社内レビュー基準
- 入力画像 / 入力動画の使い方
- 過去の成功例と失敗例

skill が「どう考えるか」だとすると、knowledge は「何を前提にするか」である。

## 保存する成果物の例

1回の相談結果は、以下のようなフォルダーで保存する。

```text
artifacts/
  2026-07-08_153000_project-shot-idea/
    request.md
    conversation.md
    prompt.md
    prompt.json
    inputs.md
    result.md
    references.md
    metadata.json
    review.md
```

### request.md

ユーザーの最初の依頼、目的、用途、制約を記録する。

### conversation.md

対話の流れを記録する。

### prompt.md

人間が読みやすい最終プロンプトを記録する。

### prompt.json

機械処理しやすい構造化データを記録する。

### inputs.md

入力画像、入力動画、参照素材、元ファイル、使用目的を記録する。

### result.md

生成結果、成功 / 失敗の判断、失敗理由、次に試す改善案を記録する。

### references.md

参照したskill、knowledge、workspace内のファイルを記録する。

### metadata.json

ユーザー、日時、想定モデル、呼び出したskill、skillの情報源、バージョン、最終確認日を記録する。

### review.md

管理者がレビューし、skill改善へ戻すためのメモを記録する。

## 改善サイクル

### 1. 用意する

会社として使うskillとknowledgeを用意する。

### 2. 使って作る

制作スタッフがChrome拡張で相談し、目的とモデルに応じたskillを使い、プロンプトと副産物を保存する。

### 3. 見返す

管理者が保存物をレビューし、品質、再現性、モデル適性、案件適合性、入力素材の妥当性、成功 / 失敗理由を確認する。

蓄積が増えた段階では、Codex、Claude Code、Gemini / Antigravity などから呼べる管理者用skillで、複数artifactを横断分析する。

### 4. 直して次へ使う

レビュー結果をskill / knowledgeへ反映し、次回以降の出力を改善する。

管理者用skillは、反映候補、根拠artifact、影響範囲、更新対象skill、confidenceを整理する。ただし、共通skillへの反映は管理者が確認してから行う。

## 管理者用skill

PromptStudio では、現場スタッフが使うプロンプト生成skillとは別に、管理者が蓄積された成果物を見て、skill改善を進めるためのskillを用意する。

現場スタッフ用skillは、制作意図を整理し、対象モデルに合ったprompt packageを作るためのもの。

管理者用skillは、保存されたartifactを分析し、共通skillとknowledgeを改善するためのもの。

初期候補。

- artifact-review: 保存された成果物を読み、意図、入力素材、モデル選択、skill呼び出し、結果、レビューを点検する
- skill-gap-analysis: 失敗例やレビューコメントから、不足しているskillや古くなったskillを見つける
- model-skill-maintenance: 公式ガイドライン、社内検証、失敗傾向をもとにモデル別skillの更新候補を出す
- review-summary: 週次 / 月次で成功例、失敗例、改善候補を管理者向けに要約する

管理者用skillが出すべきもの。

- 改善提案
- 根拠となるartifact
- 失敗 / 成功の傾向
- 対象skill / knowledge
- 想定される影響範囲
- 公式情報の再確認が必要な箇所
- confidence
- 管理者承認が必要であること

重要なのは、管理者用skillが共通skillを勝手に書き換えないことである。skill改善は制作品質に直接影響するため、提案、レビュー、承認、反映の流れを分ける。

## レビュー観点

管理者は、少なくとも以下を見る。

- 制作意図を正しく分解できているか
- 画像生成 / 動画生成の用途に合っているか
- モデルの制約に合っているか
- 使ったモデルに対して適切なskillが呼ばれているか
- 入力画像 / 入力動画 / 参照素材は目的に合っているか
- 失敗した場合、原因が次回改善できる形で記録されているか
- 不要に曖昧な表現が残っていないか
- 社内ルールや案件ルールに違反していないか
- 次回も再利用できる形になっているか
- skillに戻すべき学びがあるか

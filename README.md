# PromptStudio

PromptStudio は、広告、映像、デザイン、CG / VFX、ゲーム、アニメ、ブランド制作、SNSコンテンツなど、クリエイティブ制作に関わるチームのための **生成AIプロンプト運用基盤** です。

生成AIは、すでに制作現場の一部になり始めています。画像生成、動画生成、コンセプト設計、絵コンテ、キービジュアル、モーション案、背景美術、商品ビジュアル、広告バリエーションなど、使い道は広がっています。

しかし現場では、成果の品質が「誰がプロンプトを書いたか」に大きく依存しています。言語化が得意な人、モデルごとの癖を知っている人、英語表現に強い人だけが良い結果を出せる。逆に、優れた制作意図を持っている人でも、生成AIに伝わる形へ変換できなければ、狙った結果に届かない。

PromptStudio は、この問題を個人の努力ではなく、会社の仕組みとして解決します。

## 一言で

PromptStudio は、クリエイティブ制作における生成AI活用を、個人のプロンプト技術から、会社全体で再現・改善できる制作プロセスへ変えるツールです。

社員は Chrome 上のチャットUIで「作りたいもの」を相談します。PromptStudio は、会社共通の skill、knowledge、案件ルール、モデル別ノウハウを使い、画像生成・動画生成・広告制作・ビジュアル開発などの目的に合わせて、実際に使えるプロンプトと副産物を組み立てます。

さらに、その会話、判断、参照情報、生成設定、最終プロンプトを保存します。管理者はそれをレビューし、skill と knowledge を改善することで、次の制作品質を上げられます。

## 重要な前提

PromptStudio が解くのは、「社員がプロンプトを書けない」という単純な問題ではありません。

実際の現場では、すでに各自が ChatGPT のようなAIチャットや、画像生成・動画生成の各種モデルを使い、プロンプトを組んでいます。問題は、そこで使われている前提、コンテキスト、モデル知識、プロンプト設計skillが社員ごとに違うことです。

同じ「かっこいい商品ビジュアルを作りたい」という依頼でも、ある人は広告文脈を入れ、ある人は構図だけを指定し、ある人はモデルの癖を踏まえた英語プロンプトにし、ある人は過去案件のトーンを知らずに進めます。

この差は、単に「AIにプロンプトを書いてもらえば解決する」ものではありません。AIに渡す前提が人によって違えば、AIが生成するプロンプトもばらつきます。

PromptStudio は、プロンプト作成そのものを自動化するだけのツールではなく、**プロンプトを書く前の前提、参照するknowledge、使うskill、モデルごとの考え方、保存とレビューの流れを会社として揃える仕組み** です。

ただし、目的はプロンプトをガチガチに固定して、全員が同じような平均化された出力を出すことではありません。

むしろ逆です。各クリエイターの意図、発想、ディレクション、表現の個性を活かすために、AIに渡す土台の前提条件を揃えます。案件ルール、モデルごとの基礎知識、入力素材の扱い方、失敗しやすい条件など、最低限そろっていないと表現以前でズレてしまう部分を仕組みで補正します。

## 使い方の基本思想

仕事でクリエイティブ制作に関わる生成AIプロンプトを考えるときは、まず PromptStudio を使う。

これが基本思想です。

社員が個別にChatGPTや各種生成AIモデルへ直接相談することを否定するのではありません。むしろ、現場では今後もさまざまなAIチャット、画像生成モデル、動画生成モデル、外部サービスが使われます。

PromptStudio は、その前段に置く共通の制作ワークベンチです。

- 何を作りたいのか
- どの媒体・用途なのか
- どのモデルで生成するのか
- 入力画像や入力動画を使うのか
- 参照すべき公式ガイドラインは何か
- 社内で改善されてきたskillは何か
- そのskillはどの情報源と検証結果に基づいているのか
- 過去に何が失敗し、何が成功したのか
- 保存してレビューすべき副産物は何か

これらを整理したうえで、使うモデルに適したプロンプトへ変換します。

## 将来像: agent向けプロンプトハブ

最初の入口は Chrome拡張ですが、PromptStudio の本体は Chrome拡張そのものではありません。

将来的には、PromptStudio を **生成AIプロンプトの前提条件、skill、knowledge、保存、レビューを提供する共通ハブ** として育てます。

つまり、以下のような外部エージェントや開発環境から呼べる立ち位置を目指します。

- ChatGPT系の環境
- Codex
- Claude Code
- Gemini / Antigravity
- その他のAIエージェント、制作支援ツール、社内ツール

呼び出し口は、複数用意します。

- Chrome拡張: 現場スタッフが毎日使う対話UI
- CLI: ターミナルや自動処理から使う入口
- MCP server: Codex、Claude Code、Gemini、Antigravityなど、MCP対応エージェントから呼ぶ入口
- 管理者用agent skill: CodexやClaude Codeから蓄積artifactを分析し、skill改善案を作る入口
- Local API: Chrome拡張、社内ツール、ローカルworkspaceから呼ぶ入口
- 将来のTeam Server: チーム単位のskill、knowledge、レビュー履歴を管理する入口

どの入口から呼ばれても、PromptStudio Core が同じ判断を行います。

- 依頼意図を整理する
- 対象モデルを確認する
- 必要なskillを選ぶ
- 参照すべきknowledgeを集める
- 入力画像 / 入力動画 / 参照素材を記録する
- モデル別プロンプト、ネガティブ、設定、注意点を出す
- 成功 / 失敗とレビュー結果を保存する
- 蓄積artifactを分析し、管理者向けのskill改善案を作る

これにより、「ChatGPTで相談した人」「Codexから呼んだ人」「Claude Codeで作業している人」「Antigravityから実行した人」がいても、会社として参照する前提条件とskillを揃えられます。

さらに、管理者やAI推進担当は、蓄積された実例をCodex / Claude Code用の管理者skillで分析し、どのモデル別skillを更新すべきか、どのknowledgeが古いか、どの失敗パターンを共通ルールへ戻すべきかを確認できます。

## モデル別skillの考え方

PromptStudio は、1つのモデルだけに最適化するツールではありません。

たとえば、以下のような用途とモデルに対応できるようにしていきます。

- GPT Image 2 で画像生成する
- Nano Banana Pro で画像生成する
- Kling 3.0 で動画生成する
- Seedance 2.0 で動画生成する
- Runway Aleph / Runway系モデルで動画や映像表現を試す
- その他、今後登場する画像生成、動画生成、3D、音声、編集系モデルを使う

モデルごとに、得意な表現、苦手な指定、必要な入力情報、推奨されるプロンプト構造、入力画像・入力動画の扱い方は違います。

そのため PromptStudio では、意図とモデルに応じて必要なskillを呼び出します。

- 公式ガイドライン由来のskill
- 社内検証で改善されたskill
- 過去案件から蓄積された成功例
- 失敗パターンから作られた注意点
- skillの情報源、バージョン、最終確認日
- 入力画像 / 入力動画を使う場合のチェック項目
- モデル別のネガティブ指定や避けるべき表現

ユーザーは「このモデルでは何をどう書けばいいか」を毎回ゼロから考える必要がありません。PromptStudio が、目的とモデルに応じて必要な前提を揃えます。

## なぜ今必要か

生成AIサービスは今後さらに増え、画像、動画、3D、音声、編集、合成など、制作工程の多くに入り込んでいきます。

しかし、ツールが増えるほど、現場では「どのサービスをどう使うか」「どう指示すれば狙った表現になるか」「案件ルールをどう反映するか」という運用の難度が上がります。

このままでは、生成AIを使いこなせる一部の人だけが速くなり、組織全体の制作力にはなりません。

PromptStudio は、特定の生成AIサービスを置き換えるものではなく、複数の生成AIサービスを制作現場で使いこなすための共通レイヤーです。

## いま起きている課題

### 1. プロンプト品質が個人の前提に依存している

同じ案件、同じ素材、同じ生成AIサービスを使っても、担当者によって出力品質が大きく変わります。さらに実際には、社員ごとに使うAIチャット、画像生成モデル、動画生成モデル、参照している文脈も違います。

原因は、制作力そのものではなく、生成AIに伝えるための分解力、言語化力、モデル理解、案件文脈、会社ノウハウへのアクセス差です。

### 2. 良いノウハウが社内資産にならない

うまくいったプロンプト、失敗した指定、モデルごとの癖、案件ごとの注意点が、個人のチャット履歴やメモに閉じています。

結果として、別の社員は同じ試行錯誤を繰り返し、会社として学習できません。

### 3. AIにプロンプトを書かせても前提が揃わない

汎用AIチャットに「良いプロンプトを書いて」と頼めば、一見それらしい文章は作れます。

しかし、そのAIが会社の案件ルール、ブランド文脈、過去の成功例、モデルごとの癖、管理者のレビュー基準を知らなければ、出てくるプロンプトは担当者の入力品質に依存します。

つまり、必要なのはプロンプト生成だけではなく、AIが参照する前提を揃える仕組みです。

### 4. 案件ルールやブランド文脈が抜け落ちる

広告、ブランド、映像制作では、単に「きれいな画像」や「派手な動画」が出ればよいわけではありません。

トーン、ターゲット、禁止表現、ブランドガイドライン、納品形式、演出意図、既存素材との整合性が必要です。しかし通常のAIチャットでは、これらを毎回人間が説明しなければなりません。

### 5. 管理者が改善できる材料が残らない

最終プロンプトだけが残っても、なぜその表現になったのか、どの知識を参照したのか、どこで意図がずれたのかが分かりません。

そのため、管理者やAI推進担当がレビューしても、次回以降の改善につながりにくい。

### 6. 生成AI活用が「便利ツール止まり」になる

現場の一部メンバーだけが使いこなし、会社全体の制作プロセスには組み込まれない。

これでは、生成AIの導入効果が属人的な時短で終わり、品質管理、教育、再利用、案件横断の改善に広がりません。

## PromptStudio が解決すること

PromptStudio は、生成AIプロンプトを「その場限りの文章」ではなく、会社が育てる制作資産として扱います。

### 1. 制作意図をプロンプトに変換する

ユーザーは、専門的なプロンプト構文を知らなくても、日本語で目的を相談できます。

PromptStudio は、目的、媒体、トーン、構図、演出、時間変化、参照素材、制約条件を整理し、画像生成や動画生成に適した構造へ変換します。

### 2. 社内skillで前提条件を揃える

skill は、単なるテンプレートではありません。

画像生成の構図設計、動画生成のカメラワーク、広告表現の注意点、ブランドトーン、モデル別の得意不得意、ネガティブプロンプト、レビュー観点などを含む、会社共通の制作ノウハウです。

skill は、表現を平均化するための固定フォーマットではありません。各自のアイデアや作家性を残したまま、プロンプトを書く前に必要な前提、制約、モデル知識、確認観点の抜け漏れを補正するためのものです。

### 3. knowledgeで案件文脈を反映する

案件ごとの世界観、ブランドガイドライン、キャラクター設定、商品情報、禁止事項、過去の参考資料を knowledge として参照します。

毎回ゼロから説明するのではなく、会社や案件が持っている文脈を使ってプロンプトを組み立てます。

### 4. モデルごとの前提差を扱う

社員が使う生成AIサービスやモデルは、必ずしも1つに統一されません。

PromptStudio は、特定モデルに閉じるのではなく、モデルごとの得意不得意、入力形式、表現の癖、必要な補足情報をskillとして扱います。

これにより、どのモデルを使う場合でも、会社の文脈と制作意図をできるだけ一貫した形で渡せるようにします。

### 5. 副産物を保存してレビュー可能にする

保存するのは、最終プロンプトだけではありません。

- 最初の依頼
- 対話ログ
- 参照したskill
- 参照したknowledge
- 最終プロンプト
- ネガティブプロンプト
- 推奨生成設定
- モデル / サービス想定
- 入力画像 / 入力動画 / 参照素材
- 成功・失敗の判断
- 失敗した場合の原因候補
- 管理者レビュー用メモ

これにより、なぜそのプロンプトになったのかを後から確認できます。

### 6. レビューからskill改善へつなげる

管理者は、保存された成果物を見て、何が良かったか、何が足りなかったか、どのskillを直すべきかを判断できます。

改善されたskillは次の社員の出力に反映されます。これにより、生成AI活用が個人技ではなく、会社全体で回るPDCAになります。

将来的には、管理者がこのレビューを手作業だけで行うのではなく、Codex、Claude Code、Gemini / Antigravity などから呼べる **管理者用skill** を用意します。管理者用skillは、蓄積されたartifactを横断的に読み、成功 / 失敗の傾向、入力素材の問題、モデル選択のズレ、呼び出されたskillの不足を分析し、共通skillの改善案を出します。ただし、改善案は自動反映せず、管理者が根拠を確認して承認する運用にします。

## 導入後に変わること

| Before | After |
| --- | --- |
| プロンプト品質が担当者の前提とモデル理解に左右される | 土台の前提条件を揃え、表現の意図をモデルに伝えやすくする |
| 良いプロンプトが個人のチャット履歴に埋もれる | 成果物として保存され、再利用とレビューができる |
| 案件ルールやブランド文脈を毎回説明する | knowledgeとして参照し、プロンプトに反映できる |
| AIにプロンプトを書かせても入力品質に依存する | AIが参照する前提をskill / knowledgeとして揃えられる |
| 生成AIに強い人だけが成果を出せる | 一般の制作スタッフも会社のノウハウを使える |
| 管理者が改善点を把握しにくい | 会話、参照情報、設定、結果を見てskill改善に戻せる |
| 生成AI活用が個人の時短で止まる | 組織の制作プロセスとナレッジ資産になる |

## 各自がChatGPTや別モデルを使う状態との違い

PromptStudio は、ChatGPTのような対話UXを目指しますが、目的は汎用チャットや単なるプロンプト自動生成ではありません。

現場では、社員がそれぞれ異なるAIチャット、画像生成モデル、動画生成モデルを使うことがあります。PromptStudio は、その使い方を一律に禁止したり、1つのモデルへ強制的に統一したりするのではなく、どのモデルを使う場合でも会社の前提を渡せるようにします。

| ChatGPTなどの汎用AIチャット | PromptStudio |
| --- | --- |
| 個人の会話として完結しやすい | 会社の制作プロセスに接続する |
| 毎回コンテキストを入力する必要がある | skill / knowledge / workspaceを参照する |
| 入力者の前提が違うと出力もばらつく | 会社として揃えた前提を使える |
| 成果物の保存形式がばらばら | レビュー可能な単位で保存する |
| 管理者が改善に使いにくい | レビュー結果をskill改善に戻せる |
| モデルや用途ごとの運用が個人任せ | 用途別・モデル別の共通ノウハウを使える |

クリエイティブ制作の中で、生成AIを使ってビジュアル、映像、企画、表現案を作るすべてのチームを対象にします。

- 広告代理店
- 映像制作会社
- ポストプロダクション
- CG / VFX スタジオ
- デザイン会社
- ブランド / マーケティング部門

- 商品撮影、ファッション、ECビジュアル制作
- イベント、展示、空間演出の企画チーム

共通している課題は、生成AIそのものではなく、制作意図を安定してAIに渡し、結果を組織的に改善する仕組みが不足していることです。

## 最初のプロダクト像

最初に実装するのは、Chrome拡張として動く対話UIです。

ユーザーは、ChatGPTに近い操作感で相談できます。ただし、PromptStudio は汎用チャットではありません。会社のskill、knowledge、workspace、保存先フォルダーを扱い、制作現場で再利用・レビューできる形にします。

裏側では、Google Antigravity / Antigravity IDE をローカル実行・agent基盤として使う構成を目指します。

```text
Chrome Extension / CLI / MCP Server / Local API
  -> PromptStudio Core
    -> Skill Registry
    -> Knowledge Registry
    -> Artifact Store
    -> Review Feedback Loop
    -> Execution Adapters
      -> Google Antigravity / Antigravity IDE
      -> Other agents and model tools
```

Chrome拡張は、対話UXと設定入力に集中します。任意フォルダーへの保存やローカルworkspace操作は、Chrome拡張から直接行わず、ローカルブリッジ経由にします。

## なぜ Antigravity なのか

PromptStudio では、初期のローカル実行・agent基盤として Google Antigravity / Antigravity IDE を重視します。

理由は、単に「AI IDEだから」ではありません。PromptStudio がやりたいことは、skill、knowledge、workspace、保存物、レビューをつなぎ、各クリエイターの前提条件を揃えることです。そのためには、チャットだけではなく、ローカルファイル、プロジェクト単位の設定、agent、artifact、ブラウザ検証、将来の外部ツール連携を扱える実行基盤が必要です。

Antigravity を選ぶ理由。

- Google の公式プロダクトとして、業界導入時の安心感と説明しやすさがある
- Googleアカウントで始められ、個人向けには無料プランが用意されている
- Google AI Pro / Ultra などの有料プランで、より高い利用上限や優先アクセスへ拡張できる
- Gemini 3.5 Flash など、Googleの上位モデルと近い位置で使える
- Gemini は画像・動画・音声・PDFなどのマルチモーダル入力に対応しており、クリエイティブ制作の文脈と相性が良い
- 動画理解では、映像と音声の両方から内容を理解し、説明や質問応答、タイムスタンプ参照ができる
- Antigravity の Projects / Skills / Artifacts の考え方が、PromptStudio の workspace / skill / review artifact 設計と噛み合う
- agent が editor、terminal、browser をまたいで作業できるため、保存、検証、レビュー、改善の流れを自動化しやすい
- MCPやGoogle製開発ツール連携により、将来的にknowledge、Drive、Workspace、社内ツールへ拡張しやすい
- Windows / macOS / Linux で使えるため、制作現場のPC環境に合わせやすい

ただし、PromptStudio 全体を Antigravity 専用に固定しません。Antigravity は現時点で有力な実行基盤ですが、保存形式、skill、knowledge、レビューPDCAは独立した資産として設計します。

現時点の注意点もあります。Antigravity の利用上限や提供条件はプランによって変わり、容量は常に保証されるものではありません。また、Googleのヘルプでは Antigravity のプロンプトと指示は英語のみ対応とされています。PromptStudio では、日本語の制作相談を英語プロンプトやモデル別指示へ変換する層を用意する前提で設計します。

詳細は [docs/09-why-antigravity.md](docs/09-why-antigravity.md) にまとめています。

## MVPで作るもの

1. Chrome拡張のサイドパネルで対話できる
2. skill、knowledge、workspace、成果物保存先を設定できる
3. 画像生成 / 動画生成 / 広告制作 / ビジュアル開発など、目的別にプロンプトを組み立てられる
4. GPT Image 2、Nano Banana Pro、Kling 3.0、Seedance 2.0、Runway系など、モデル別skillを呼び出せる
5. 会話、生成プロンプト、参照情報、設定、入力画像 / 入力動画、副産物をレビュー可能な形で保存できる
6. 管理者が保存された実例を見て、skill改善に使える

Googleアカウントログイン、権限管理、チーム単位のダッシュボードは重要ですが、MVPでは「誰が作った成果物かを記録できる設計」に留め、後から拡張します。

## 保存される成果物

1回の相談を、1つのレビュー可能な成果物として保存します。

```text
artifacts/
  2026-07-08_153000_campaign-visual-idea/
    request.md
    conversation.md
    prompt.md
    prompt.json
    inputs.md
    result.md
    references.md
    metadata.json
    review.md
```

この構造により、現場はプロンプトを再利用でき、管理者は改善点を見つけられます。

特に重要なのは、プロンプトだけでなく、入力に使った画像・動画・参照素材、生成結果、成功/失敗の判断、改善コメントまで残すことです。これが次のskill改善の材料になります。

## ビジネス価値

### 前提条件の安定化

プロンプト作成をガチガチに規格化するのではなく、案件ルール、モデル知識、入力素材の扱い方、レビュー観点など、表現の土台になる前提条件を揃えます。

### 制作スピードの向上

毎回ゼロから考える時間を減らし、用途ごとに必要な要素を対話で素早く整理できます。

### 教育コストの削減

新人や生成AIに慣れていないメンバーでも、会社のノウハウを使いながら出力できます。

### ナレッジの蓄積

良いプロンプト、失敗例、案件ごとの注意点を、個人メモではなく会社の資産として残せます。

### 管理者による改善

実際の利用データをもとにskillを改善できるため、使えば使うほど会社全体の生成AI活用品質が上がります。

管理者用のCodex / Claude Code skillを整備すれば、個別レビューだけでなく、一定期間のartifactをまとめて分析し、「失敗が多いモデル」「更新が必要なskill」「公式ガイドライン確認が必要な箇所」「新人がつまずきやすい前提」を把握できます。

## 現時点の設計方針

- Chrome拡張はUXと設定入力に集中する
- 任意フォルダー保存やローカルworkspace操作は、Chrome拡張から直接ではなく、ローカルブリッジ経由にする
- PromptStudio Core を中心に置き、Chrome拡張、CLI、MCP server、Local API から同じskill / knowledge / artifactを呼べるようにする
- Google Antigravity / Antigravity IDE への依存は抽象化し、将来ほかの実行エンジンにも差し替えられるようにする
- skill は「プロンプトの作法」だけでなく、モデル別制約、制作レビュー観点、案件コンテキストを含める
- 管理者用skillは、蓄積artifactの分析と改善提案を担当し、共通skillの自動書き換えは行わない
- knowledge は、会社・案件・ブランド・制作ルールの共有文脈として扱う
- モデル別skillは、公式ガイドラインと社内検証で改善された知見の両方を扱う
- 成果物は後からレビューしやすい単位で保存する

## 目指す状態

PromptStudio が目指すのは、生成AIを一部の得意な人だけが使う状態ではありません。

クリエイティブチーム全体が、会社のノウハウを使って前提条件を揃えながら、それぞれの意図や表現の違いを活かしたプロンプトを作り、結果を保存し、レビューし、次の制作に活かせる状態です。

生成AIを「便利な外部ツール」から、「会社の制作力を拡張する運用基盤」へ変えることが、PromptStudio の目的です。
